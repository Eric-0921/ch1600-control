# COM13 真实设备通信记录

时间：2026-05-21

本次测试使用 Python + pyserial 直接和设备通信。测试前已确认 `Debug/SystemConfig.xml` 中配置为：

- `MODEL = CH-1600`
- `COMWAY = 串口`
- `COMMSET = COM13`
- `BADURATE = 115200`
- `DATATYPE = 高速数据`
- `SPEED = 20组/s`

串口枚举结果：


Python 环境：

## 测试 1：原厂高速 20 组/s

发送顺序：

```text
TX 'DATAC>' HEX=44 41 54 41 43 3e
TX 'TYPE>'  HEX=54 59 50 45 3e
TX 'FAST2>' HEX=46 41 53 54 32 3e
TX 'DATAC>' HEX=44 41 54 41 43 3e
```

观测结果：

```text
RX_AFTER_STOP len=0
RX_TYPE len=2 repr=b'M\n'
RX_FAST2 len=205
RX_FAST2_LINES count=15
```

高速返回样例：

```text
#+0000.1605/000/+0283>
#+0000.1496>
#+0000.1600>
#+0000.1569>
#+0000.1589>
#+0000.1572>
#+0000.1588>
#+0000.1570>
#+0000.1580>
#+0000.1574>
```

说明：

- `FAST2>` 对真实 CH-1600 有效。
- 第一行出现普通三段帧 `磁场/频率/温度`，后续变为高速纯磁场帧。
- 高速纯磁场帧形态为 `#{field}>\n`，例如 `#+0000.1496>\n`。
- `TYPE>` 实测返回 `M\n`，不是反编译代码中处理的 `WEAK\n` / `NORM\n`。这一点需要继续确认，不应直接按原代码假设。

## 测试 2：普通采集 DATA?>

发送顺序：

```text
TX 'DATAC>' HEX=44 41 54 41 43 3e
TX 'DATA?>' HEX=44 41 54 41 3f 3e
TX 'DATAC>' HEX=44 41 54 41 43 3e
```

观测结果：

```text
RX_DATAQ len=184
RX_DATAQ_LINES count=8
```

普通返回样例：

```text
#+0000.1357/000/+0283>
#+0000.1357/000/+0283>
#+0000.1357/000/+0283>
#+0000.1357/000/+0283>
#+0000.1354/000/+0283>
#+0000.1354/000/+0283>
#+0000.1350/000/+0283>
#+0000.1350/000/+0283>
```

说明：

- `DATA?>` 对真实 CH-1600 有效。
- 普通帧形态为 `#{field}/{freq}/{temp_x10}>\n`。
- 样例温度字段 `+0283`，按原厂软件逻辑除以 10，即 `28.3 ℃`。
- 样例频率字段为 `000`。

## 对反编译结论的验证

| 结论 | 实测结果 |
| --- | --- |
| 停止命令 `DATAC>` | 有效，停止后无继续输出 |
| 普通启动命令 `DATA?>` | 有效 |
| CH-1600 高速 20 组/s 使用 `FAST2>` | 有效 |
| 普通帧包含磁场/频率/温度 | 已验证 |
| 高速帧只包含磁场 | 已验证，但启动后首行可能残留/过渡为普通帧 |
| 温度字段需要除以 10 | 与样例 `+0283 => 28.3 ℃` 一致 |

## 后续建议

- Python 驱动的高速解析器需要容忍启动后的第一行三段普通帧，然后进入纯高速帧。
- `TYPE>` 对 CH-1600 的 `M\n` 返回需要单独建模或忽略。
- 若实现高速 20 Hz，优先使用原厂实测有效的 `FAST2>`。
- 每次测试结束都发送 `DATAC>`，避免设备继续流式输出。
