# m1600 改进路线图

> 基于 DataReader2 逆向工程发现报告制定
> 制定时间：2026-05-17

---

## 1. 概述

本文档根据对原厂上位机软件 DataReader2.exe 的逆向分析结果，制定了 m1600 项目的分阶段改进计划。所有改进项按 **P0（紧急）/ P1（重要）/ P2（可选）** 三级优先级划分，每个改进项包含目标、实现思路、验收标准和预估工作量。

### 当前状态速览

| 维度 | m1600 现状 | DataReader2 参考 | 差距 |
|------|-----------|-----------------|------|
| 采样速率 | GUI 显示多档，底层未发指令 | 6 档 `FASTxxx>` | **需补齐** |
| 单位显示 | 仅 mT | 5 种单位实时切换 | 需补齐 |
| 阈值报警 | 仅设置参数 | NG/OK + 报警灯 | 需补齐 |
| 数据表格 | ❌ | DataGridView | 需添加 |
| 调试窗口 | 日志页面 | Hex/ASCII 收发 | 需加强 |
| 外部集成 | ❌ | NamedPipe + SendKeys | 可添加 |
| 数据导出 | 图表图片 | Excel + TXT | 可添加 |
| 文件策略 | 单文件 CSV | 自动新建/停止 | 可优化 |

## 2. P0 优先级（紧急，建议立即实施）

### 2.1 实现 `FASTxxx>` 采样速率指令

**目标**：让 m1600 真正控制设备的高速采样硬件模式，而非软件降采样。

**背景**：逆向发现 DataReader2 通过 `FAST020>` ~ `FAST300>` 系列指令激活设备的不同采样速率。当前 m1600 的 GUI 虽显示 20/50/100/200/200+ Hz 模式，但底层从未发送这些指令。

**实现思路**：
1. 在 `core/commands.py` 的 `CommandType` 枚举中新增 `SET_SAMPLE_RATE` 成员
2. 在 `instruments/ch1600_driver.py` 中实现 `set_sample_rate(hz: int)` 方法，映射关系：
   | 速率 | 指令 | 适用设备 |
   |------|------|----------|
   | 常速 | `DATA?>` | 全部 |
   | 20Hz | `FAST2>`（一维）/ `FAST020>`（其他） | 一维用简写 |
   | 50Hz | `FAST050>` | 全部 |
   | 100Hz | `FAST100>` | 全部 |
   | 200Hz | `FAST200>` | 全部 |
   | 300Hz | `FAST300>` | 全部 |
3. 在 `core/command_service.py` 的 `_execute()` 中增加路由分支
4. 在 `app/gui.py` 的采集模式选择下拉框绑定实际指令发送
5. 更新 `app/config_io.py` 的 `ACQ_MODE_TABLE`，增加 `command` 字段存储对应指令
6. 在 `CH1600StreamWorker` 中，启动流时根据当前模式发送正确的启动指令

**验收标准**：
- [ ] 选择不同采集模式后，串口实际发送对应的 `FASTxxx>` 指令
- [ ] 设备返回的数据帧速率与选择模式匹配（可用逻辑分析仪或串口监视验证）
- [ ] 配置文件正确保存当前采样模式，重启后恢复

**预估工作量**：4-6 小时

### 2.2 单位换算显示系统

**目标**：支持 mT / G / Oe / A/m / mGs 五种单位的实时切换显示。

**背景**：DataReader2 有完整的 5 种单位换算矩阵。当前 m1600 只显示 mT。

**实现思路**：
1. 在 `data/circular_buffer.py` 或 `app/gui.py` 中增加单位换算层
2. 换算矩阵（与 DataReader2 一致）：
   | 单位 | 系数 |
   |------|------|
   | mT | ×1 |
   | G | ×10 |
   | Oe | ×10 |
   | A/m | ×79.577 |
   | mGs | ×10000 |
3. GUI 大字体数值显示区域增加单位下拉框
4. CSV 记录仍保存原始 mT 值（避免数据歧义），显示层做换算
5. 配置文件保存用户首选单位

**验收标准**：
- [ ] 5 种单位下拉切换后，显示数值实时更新
- [ ] 换算结果与 DataReader2 一致（取相同测试数据交叉验证）
- [ ] CSV 文件中的数据始终为原始 mT 值

**预估工作量**：3-4 小时

## 3. P1 优先级（重要，建议近期实施）

### 3.1 阈值报警可视化

**目标**：实现 NG/OK 实时报警状态显示，并可选择性地支持外接报警灯。

**背景**：DataReader2 有完整的阈值判断逻辑（开区间/闭区间/ABS），并用红/绿背景色直观显示状态。

**实现思路**：
1. 在 `app/gui.py` 的实时数据页面增加阈值状态区域
2. 状态显示逻辑（与 DataReader2 一致）：
   - 开区间 + 值在 (下限, 上限) 内 → NG（红色）
   - 开区间 + 值在区间外 → OK（绿色）
   - 闭区间 + 值在 [下限, 上限] 内 → OK（绿色）
   - 闭区间 + 值在区间外 → NG（红色）
3. ABS 模式：先取绝对值再判断
4. 可选：增加蜂鸣器/报警灯串口控制（通过第二个串口发送 Modbus RTU）
5. 报警灯控制字节序列（来自逆向源码）：
   ```python
   RED_LAMP = bytes([1, 5, 0, 16, 255, 0, 141, 255])
   BEEP_ON = bytes([1, 5, 0, 161, 255, 0, 221, 216])
   GREEN_LAMP = bytes([1, 5, 0, 1, 255, 0, 221, 250])
   BEEP_OFF = bytes([1, 5, 0, 161, 0, 0, 156, 40])
   LAMP_OFF = bytes([1, 5, 0, 4, 0, 0, 140, 11])
   ```

**验收标准**：
- [ ] 阈值超限后 GUI 状态区域变色（红/绿）
- [ ] 开区间/闭区间/ABS 三种模式正确切换
- [ ] （可选）配置第二个串口后，报警灯正确响应

**预估工作量**：6-8 小时（不含报警灯硬件联调）

---

### 3.2 实时数据表格

**目标**：在实时数据页面增加可展开的数据表格，展示每帧原始数据。

**背景**：DataReader2 使用 DataGridView 展示每帧的序号、X/Y/Z 值、温度、频率、时间戳。

**实现思路**：
1. 在 `app/gui.py` 实时数据页面增加 `QTableWidget` 或 `QTableView`
2. 表头：序号 | 磁场值(mT) | 频率(Hz) | 温度(°C) | 时间戳
3. 数据来源：`CircularBuffer` 的追加事件
4. 限制显示行数（如最新 1000 行），超出时滚动移除旧行
5. 支持选中行高亮，与图表联动（点击表格行，图表跳转到对应时间点）

**验收标准**：
- [ ] 数据表格与图表实时同步
- [ ] 表格支持选中/复制/导出
- [ ] 性能：100Hz 采样下不卡顿

**预估工作量**：4-6 小时

---

### 3.3 原始串口数据调试窗口

**目标**：增加专门的调试页面，显示原始串口收发数据的 Hex/ASCII 格式。

**背景**：DataReader2 有专门的调试模式，支持 Hex/ASCII 切换和手动发送任意指令。

**实现思路**：
1. 在 `app/gui.py` 增加"调试"页面（与"日志"页面并列或替代）
2. 双区域布局：
   - 上部：接收数据区（Hex/ASCII 切换）
   - 下部：发送区（输入框 + 发送按钮）
3. 从 `CH1600Driver` 的原始读取字节中分流一份到调试窗口
4. 发送区支持：
   - 直接输入 ASCII 文本发送（自动追加 `\r`）
   - Hex 模式输入（如 `44 41 54 41 3F 3E`）
   - 快捷按钮：`DATA?>`, `DATAC>`, `DATAS>`, `ZERO>`, `FAST020>` 等
5. 颜色编码：发送绿色，接收蓝色，错误红色

**验收标准**：
- [ ] 所有串口收发数据实时显示
- [ ] Hex/ASCII 切换正确
- [ ] 手动发送指令设备正确响应

**预估工作量**：4-5 小时

---

### 3.4 数据导出功能（Excel/TXT）

**目标**：支持将采集数据导出为 Excel 和制表符分隔文本。

**背景**：DataReader2 支持通过 Microsoft.Office.Interop.Excel 自动化导出。

**实现思路**：
1. Excel 导出：
   - 使用 `openpyxl` 或 `xlsxwriter` 库
   - 表头格式：宋体、12号、居中、带边框
   - 数值前加单引号 `'` 防止 Excel 自动转换（DataReader2 的做法）
   - 导出时显示进度条
2. TXT 导出：
   - 制表符分隔（与 DataReader2 兼容）
   - UTF-8 BOM 编码
3. 在 GUI 的"实时数据"页面增加"导出"按钮
4. 导出范围：当前 CSV 文件的全部数据，或用户选定时间范围

**验收标准**：
- [ ] Excel 文件可用 Excel/WPS 正常打开
- [ ] TXT 文件可用 DataReader2 的"数据回看"功能打开
- [ ] 大数据量（>10万行）导出不 OOM

**预估工作量**：3-4 小时

## 4. P2 优先级（可选，长期规划）

### 4.1 自动文件轮转

**目标**：CSV 记录支持按文件大小上限自动新建文件或停止。

**背景**：DataReader2 支持"达到数据上限后自动新建文件或停止采集"。

**实现思路**：
1. 在 `data/recorder.py` 的 `CH1600Recorder` 中增加文件大小监控
2. 配置参数：
   - `max_file_size_mb`: 单个文件上限（默认 100MB）
   - `max_file_rows`: 行数上限（默认 100,000 行）
   - `rollover_strategy`: `"new_file"`（新建）/ `"stop"`（停止）
3. 文件命名：`ch1600_YYYYMMDD_HHMMSS_2.csv`, `_3.csv`...
4. 在 GUI 的状态栏显示当前文件大小和预计剩余时间

**验收标准**：
- [ ] 文件达到上限后自动新建
- [ ] 新建文件时表头重新写入
- [ ] 状态栏正确显示文件大小

**预估工作量**：3-4 小时

---

### 4.2 数据回看功能

**目标**：支持加载历史 CSV/TXT 文件，用 pyqtgraph 回放波形。

**背景**：DataReader2 的 `DataReview.cs` 支持历史数据回放，可追加多个文件。

**实现思路**：
1. 在 GUI 增加"数据回看"页面
2. 文件选择对话框，支持多选
3. 用 `pandas` 或 `csv` 模块读取文件
4. pyqtgraph 绘制历史曲线
5. 支持时间轴缩放、平移
6. 支持追加文件到当前视图

**验收标准**：
- [ ] 可正确加载 m1600 生成的 CSV
- [ ] 可正确加载 DataReader2 生成的 TXT
- [ ] 时间轴显示正确

**预估工作量**：6-8 小时

---

### 4.3 NamedPipe / ZMQ 外部集成

**目标**：支持外部程序通过 IPC 控制 m1600 或接收实时数据。

**背景**：DataReader2 支持 NamedPipe 和 SendKeys 两种外部集成方式。

**实现思路**：
1. NamedPipe（Windows 专用）：
   - 使用 `win32pipe` 或 `multiprocessing.connection` 实现
   - 支持命令：启动、停止、切换模式、获取状态
2. ZMQ（跨平台）：
   - 使用 `pyzmq` 库
   - PUB 端口实时广播数据
   - REP 端口接收控制命令
3. SendKeys（Windows 专用）：
   - 使用 `pyautogui` 或 `pywinauto`
   - 将当前读数发送到焦点窗口

**验收标准**：
- [ ] 外部 Python 脚本可通过 ZMQ 控制 m1600
- [ ] 数据实时广播延迟 < 10ms

**预估工作量**：8-12 小时

---

### 4.4 图表交互增强

**目标**：增加用户可调的图表参数。

**背景**：DataReader2 的 ZedGraph 支持曲线颜色、线宽、长度限制滑块等自定义。

**实现思路**：
1. 曲线颜色：右键菜单或配置面板选择 CH1-CH4 颜色
2. 线宽：TrackBar 滑块调节（1-5px）
3. 历史点数：滑块调节环形缓冲区容量（1000-20000 点）
4. Y 轴范围：手动输入或自动模式切换
5. 保存图表配置到 `config.json`

**验收标准**：
- [ ] 颜色选择后实时生效
- [ ] 配置重启后恢复

**预估工作量**：4-6 小时

---

### 4.5 多设备模型支持

**目标**：扩展解析层支持二维/三维高斯计和磁通计。

**背景**：DataReader2 支持 6 种设备模型，每种有独立的解析函数。

**实现思路**：
1. 在 `instruments/ch1600_driver.py` 的 `parse_stream_frame()` 中增加模型参数
2. 实现各模型的解析分支：
   - 一维：标准 `#值/频率/温度>`
   - 二维：`X/频率/温度;Y/频率/温度`
   - 三维：`X/频率/温度;Y/频率/温度;Z/频率/温度`
   - 磁通计：去除 `\0` 后解析
3. GUI 增加设备型号选择下拉框
4. 数据层扩展为多维数组

**验收标准**：
- [ ] 各模型数据正确解析
- [ ] GUI 正确显示多通道

**预估工作量**：12-16 小时（涉及数据层和 UI 层大改）

## 6. 验收流程

每个改进项完成后，按以下流程验收：

1. **代码审查**：检查是否符合 m1600 的 6 层架构和编码风格
2. **单元测试**：新增/修改的功能有无对应的测试
3. **集成测试**：与现有 GUI、Worker、Driver 的兼容性
4. **设备联调**：连接真实 CH-1600 设备验证功能
5. **文档更新**：更新 README/AGENTS.md 中相关说明
